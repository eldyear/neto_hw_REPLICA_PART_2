# Домашнее задание к занятию «Репликация и масштабирование. Часть 2» - Элдияр Акматов

## Задание 1

**Условие** Опишите основные преимущества использования масштабирования методами:

   * активный master-сервер и пассивный репликационный slave-сервер;
   * master-сервер и несколько slave-серверов;

1. Активный master-сервер и пассивный репликационный slave-сервер - это схема обеспечения отказоустойчивости, а не распределения нагрузки. Пассивный slave здесь обычно не принимает запросы на чтение от пользователей, он просто копирует данные.
   * Если master сгорает, пассивный slave, имея точную копию данных, за секунды становится новым мастером.
   * Делать бекапы с активного master сервера опасно, вместо этого можем делать бэкапы с пассивного слейва, вообще не трогая боевой мастер.
2. Master-сервер и несколько slave-серверов - Это уже масштабирование на чтение.
   * Данные больше читаются пользователями чем пишутся по этому мы можем распределить запросы на чтение по пяти или десяти slave-серверам.
   * Географически выгодно распределить slave-серверы, раскидать по разным регионам или странам, чтобы пользователи из этих регионов читали данные с минимальным пингом.
   * Еще можем нагружать максимально один slave чтобы вытащить огромный отчет аналитикам, master и остальные slave сервера останутся в штатном состоянии.
  

## Задание 2

**Условие** Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:

    пользователи,
    книги,
    магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.

План шардирования базы данных
1. Архитектурный принцип разделения

Система будет состоять из двух независимых кластеров баз данных (применил вертикальное шардирование):

    Кластер А (Сервис Магазинов): Сюда вертикально выделена таблица магазины. Так как объем данных небольшой, горизонтальное деление не требуется.

    Кластер Б (Сервис Пользователей и Книг): Сюда уходят таблицы пользователи и книги. Из-за огромного объема данных этот кластер шардируется горизонтально.

1. Принцип горизонтального шардирования (Разбивка строк)

    Таблица пользователи: Делится на 2 шарда по принципу Hash(user_id) % 2:

        Шард 1: Хранит пользователей с нечетными id.

        Шард 2: Хранит пользователей с четными id.

    Таблица книги: Делится на 2 шарда по алфавитному признаку:

        Шард 1: Книги, названия которых начинаются на буквы от А до М.

        Шард 2 : Книги, названия которых начинаются на буквы от Н до Я.

2. Режимы работы серверов (Отказоустойчивость)

Оставлять шарды без копий нельзя: если упадет сервер (Шард 1), мы потеряем ровно половину пользователей приложения. Поэтому каждый шард и каждая база будут работать в режиме Master-Slave:

    Master: Принимает все операции записи.

    Slave: В реальном времени копирует данные с Мастера и служит для тяжелых запросов на чтение.
